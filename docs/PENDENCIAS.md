# Pendências de robustez

Itens identificados que precisam ser resolvidos antes de abrir o sistema para usuários reais e dinheiro real. Em ordem de risco, não de data.

1. ~~**Concorrência no agendamento**~~ — ✅ resolvido em 2026-06-19: índice único parcial `agendamentos_medico_horario_ativo_uidx` (medico_id+horario, só para status ativo) impede double-booking no banco. Frontend trata erro `23505` e avisa o tutor para escolher outro horário.
2. ~~**Expiração automática de reserva**~~ — ✅ resolvido em 2026-06-19: `pg_cron` roda a função `expirar_reservas_vencidas()` a cada 5 min, marcando como `expirado` quem passou de `expira_em` sem pagar.
3. ~~**Nunca confiar em preço vindo do cliente**~~ — ✅ resolvido em 2026-06-19: trigger `agendamentos_validar_valor` recalcula `valor` a partir de `medico_exames.preco` antes de gravar, ignorando o que vier do navegador.
4. **Webhook do gateway como única fonte de verdade do pagamento** — ainda não implementado. O frontend hoje cria o `agendamento` real (status `reservado`) mas não escreve em `pagamentos` (RLS bloqueia isso de propósito). Falta integrar Pagar.me/outro gateway com webhook server-side (Edge Function) que confirme o pagamento e mude o status. ⚠ **Stand-in temporário criado em 2026-06-21:** o admin agora tem um botão "Marcar como pago" na aba Agendamentos, que muda manualmente o status de `reservado` para `pago` — só existe para viabilizar testar o painel do médico (que só revela dados de tutor/pet quando o agendamento está pago/confirmado) enquanto o gateway real não existe. **Isso precisa ser desativado/restrito quando o pagamento real entrar em produção** — não pode continuar sendo possível "decretar" um pagamento sem ele ter ocorrido de fato.
5. **Backups / Point-in-Time Recovery no Supabase** — o plano Free não tem PITR; revisar ao migrar para plano pago, antes de ter dados reais de pessoas.
6. **Monitoramento de erros e logs em produção** (ex: Sentry) — para saber quando algo quebra sem depender de reclamação de usuário.
7. **LGPD** — política de privacidade real, consentimento explícito para guardar endereço residencial e dados de pet/tutor, mecanismo de exclusão de conta/dados. ⚠ Parcialmente avançado em 2026-06-20: cadastro de médico agora exige aceite explícito de 3 termos (funcionamento do Animagem, uso de dados para divulgação, forma de pagamento/repasse) com versão e data registradas em `medico_onboarding`. **Mas o texto desses termos foi escrito por mim (IA), não por um advogado — precisa de revisão jurídica antes de valer como proteção real para a empresa.** Falta ainda: política de privacidade para tutores, consentimento de dados de pet/tutor, mecanismo de exclusão de conta/dados.
8. **Monetização por publicidade** — estrutura de dados e UI já criadas (ver seção detalhada abaixo), venda ainda não ativada. Falta: tela de admin para cadastrar patrocinador sem SQL, modelo de cobrança, critério de quem pode anunciar, limite de slots por página.
9. **E-mail transacional de confirmação de agendamento** — hoje não existe. Quando o pagamento for confirmado (mesmo evento da pendência 4, via webhook do gateway), o sistema deve enviar e-mail automático para **tutor e médico**, com os dados do agendamento (exame, data/hora, pet, endereço para o médico; exame, data/hora, médico para o tutor). Os únicos e-mails que o projeto dispara hoje são os de autenticação do Supabase (confirmação de cadastro/recuperação de senha), que usam o serviço de e-mail interno do Supabase — com limite baixo por hora no plano gratuito e não recomendado para e-mail transacional de produto. Vai precisar de um provedor dedicado (ex: Resend, Postmark) chamado por uma Edge Function disparada pelo mesmo webhook de pagamento.
10. ~~**Médico self-service: área de atendimento, exames e dados bancários**~~ — ✅ resolvido em 2026-06-20: médico agora cadastra área de atendimento real via Estado/Cidade (API do IBGE) + Bairro, cadastra/edita seus próprios exames com preço diferenciado por dia de semana/fim de semana/noturno, e cadastra dados bancários (PIX/conta) para repasse — guardados em tabela própria (`medico_dados_bancarios`) com RLS restrita ao próprio médico e ao admin, nunca exposta no portal público.
11. ~~**Cadastro de médico sem validação de documentos**~~ — ✅ resolvido em 2026-06-20: cadastro de médico agora exige CRMV, telefone, upload de documento (CRMV ou identidade, guardado em bucket privado `documentos-medicos` no Storage, só o próprio médico e o admin conseguem abrir) e aceite explícito de 3 termos. O admin vê tudo isso na aba Médicos antes de aprovar (botão "Ver doc."). Continua **pendente revisão jurídica do texto dos termos** (ver pendência 7). ✅ Reforçado em 2026-06-20: a trava agora também existe **no banco**, não só na tela de cadastro — trigger `trg_validar_ativacao_medico` bloqueia `medicos.ativo=true` se não existir uma linha em `medico_onboarding` com `status='aprovado'`, mesmo que alguém tente ativar via SQL direto ou outra rota que não seja o botão "Aprovar" do admin.
12. **Validação de cartão de crédito ainda é só de formato** — corrigido em 2026-06-20 o bug em que o número do cartão nunca era validado de fato (erro de precedência de operador no código). Agora número, nome, CPF (com dígito verificador), validade e CVV são obrigatórios e validados no formato — mas isso é só validação de formulário; **não há gateway de pagamento real ligado ainda** (mesma pendência 4). Em produção, esses dados devem ir direto para o gateway (tokenização), nunca passar pelo nosso backend.
13. ~~**Painel do médico sem dados reais (dashboard e histórico mockados)**~~ — ✅ resolvido em 2026-06-21: Dashboard e nova aba "Histórico" (extrato de atendimentos e valores) agora consultam dados reais (`agendamentos`+`pets`+`profiles`+`agendamento_exames`), respeitando a regra de só mostrar tutor/pet quando o agendamento está pago/confirmado. Aba Agendamentos do admin também passou a ser real.

**Contexto:** essas pendências foram levantadas depois da Fase 0 (repo GitHub + schema Supabase com RLS + deploy estático), quando ficou claro que o frontend ainda não fala de fato com o banco — é a Fase 1 que vai expor esses riscos na prática.

**Nota de infraestrutura (2026-06-20):** o deploy migrou do Netlify para o **Cloudflare Pages** (https://animagem.pages.dev), porque a conta do Netlify ficou sem créditos e bloqueou deploys de produção sem aviso — vários ajustes feitos numa sessão não foram publicados de fato, mesmo eu confirmando "publicado". Cloudflare Pages está conectado ao GitHub (`asouzacoelho/animagem`, branch `main`, publish directory `public`), com deploy automático a cada push, sem esse risco de créditos.

## Monetização por publicidade (estrutura criada, venda NÃO ativada)

Em 2026-06-19, a pedido do André, criamos a estrutura para vender espaço patrocinado no portal — pensando em ROI desde já, mesmo sem vender nada ainda.

- **Schema:** tabela `patrocinadores` (nome, descrição, link, cidade, `ativo`, `ordem`). RLS: leitura pública só de quem está `ativo=true`; escrita só admin. Seed de demonstração: "Animaltec" (Campinas), marcado como exemplo.
- **Frontend:** bloco "Conteúdo patrocinado" no portal (`public/index.html`), claramente rotulado como patrocínio, entre os resultados de busca e o rodapé de estatísticas. Hoje mostra 1 patrocinador ativo; o componente já suporta múltiplos (`order by ordem`).
- **Por que essa estrutura, não outra:** rótulo "Conteúdo patrocinado" visível evita parecer anúncio enganoso e problema de confiança num portal de saúde de pet — isso foi decisão deliberada, não só estética.
- **O que falta para vender de fato (quando decidirem ativar):**
  - Tela de admin para cadastrar/editar patrocinador sem precisar de SQL (hoje só dá pra inserir via banco).
  - Modelo de cobrança: mensalidade fixa por slot é o mais simples de operar agora (sem precisar de tracking de clique/impressão); CPM/CPC exigiria instrumentação extra.
  - Definir quem pode anunciar: negócios locais relacionados a pet em Campinas/região (clínicas, pet food, laboratórios) — evitar anúncio genérico que não tenha relação com o público.
  - Limite de quantos slots por página, para não comprometer a experiência do tutor.
- **Por que vale considerar para o ROI do projeto:** é receita não-dependente de comissão de agendamento — relevante porque hoje a comissão está em 0% (fase MVP) e o volume de transações ainda é baixo; patrocínio pode gerar caixa antes do marketplace atingir volume.
